What is ReDOS and what part do ‘Evil Regex’ play?

According to Wiedman (N.D) ReDoS stands for Regular expression Denial of Service and exploits regular expression entering extreme situations for the program to make it process for a long time.

According to Larson (2018), regular expressions are used in string validation as they process strings in certain format like patterns.

Wiedman (N.D) relates these as regex can overload a network, system or database by entering maliciously crafted input.

What are the common problems associated with the use of regex? How can these be mitigated?

Regex is not a very intuitive formal language and it causes many problems among people to understand it due to it being abstract and extensive symbolic notation (Ade-Ibijola, 2018). According to Larson (2018) it is also problematic to programmers as it can easily incur errors due to long strings and character sequences.

Ade-Ibijola (2018) introduces games, simulations and visualizers as a method to learn it in a more intuitive way. However, he also proposes algorithms that convert the regular expressions (REs) into short-text responses and multiple-choice questions.

Larson (2018) on the other hand describes how the ACRE (Automatic Checking of Regular Expressions) engine can help flag issues in REs creation with 11 automatic checks. Nevertheless, he warns about false alarms and a few expressions that might trigger these.

How and why could regex be used as part of a security solution?

In the most general cases, regex could be used to filter out unwanted requests preventing injections for instance at API level. When trying to tamper the header of an API request, some attacks involve the creation of encoded characters to allow for Booleans that are always true to happen in html format. (Madden, 2020).

Rathnayake and Thielecke (2014) explore how regex might also be used with a complementary tool like RXXR using backtracking to mitigate denial of service (DoS). Alongside with a different author, they expanded on the topic by explaining that this could be achieved through a parser that can transform the regex input into an abstract syntax tree (AST). This one, with a series of rules based on complex logic, can filter out suspicious requests (Kirrage, Rathnayake and Thielecke, 2013).
