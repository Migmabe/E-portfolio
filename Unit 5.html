Unit activities

According to IBM (2018), Cyclomatic Complexity was created by Thomas McCabe to determine the stability and level of confidence in a program. It measures the number of independent paths in the program, often called as decision points. The deeper this goes and more branches it has, the more complex the program is said to be.

With this introduction, let me gauge the accuracy and relevancy of Cyclomatic Complexity nowadays and whether it still remains valid to test code. Lenarduzzi, Kilamo and Janes (2023), did an empirical study with some background research, comparing Cyclomatic and Cognitive Complexity. The latter, relies on other fundamental laws and in theory is more adaptive than Cyclomatic. This is because not every decision point should count the same in the complexity scale, as some points are too hard or too easy for humans to understand. Theory supports this with the results they obtained on the test, as Cognitive Complexity yielded more accurate results than Cyclomatic to categorise 12 Java classes with more than 200 students taking the survey.

However, on the other hand, Wijendra and Hewagamage (2021) state that Cyclomatic and Cognitive complexity are not related or complementary to each other as the latter is based on statements beyond computability, like human knowledge which is quite subjective.

The perfect combination of Cyclomatic and Cognitive is the approach developed in an Object-Oriented Programming (OOP) application in Waweru (2013). In broad terms, his research thesis removes the human knowledge from the Cognitive Complexity but adds different weighing to different components like nesting level, inheritance and control structure.

Determining code complexity is essential in the development of software (Newcombe et al., 2015), as the higher the complexity, the higher the risk of costly human and machine errors. This is extremely relevant in the software industry, proof of that is the enhanced Amazon Web Services, Inc. (2020) complexity detection tool Amazon CodeGuru Reviewer.

Specific to this module, assessing the complexity of our code will help me scan the proposed system, easing the risk mitigation stages. Furthermore, complexity will have a big impact on testing to adapt the checks and understand the output.

def equivalence_partition(iterable, relation):
"""Partitions a set of objects into equivalence classes

Args:
	iterable: collection of objects to be partitioned
	relation: equivalence relation. I.e. relation(o1,o2) evaluates to True
		if and only if o1 and o2 are equivalent

Returns: classes, partitions
	classes: A sequence of sets. Each one is an equivalence class
	partitions: A dictionary mapping objects to equivalence classes
"""
classes = []
partitions = {}
for o in iterable:  # for each object
	# find the class it is in
	found = False
	for c in classes:
		if relation(next(iter(c)), o):  # is it equivalent to this class?
			c.add(o)
			partitions[o] = c
			found = True
			break
	if not found:  # it is in a new class
		classes.append(set([o]))
		partitions[o] = classes[-1]
return classes, partitions


def equivalence_enumeration(iterable, relation):
"""Partitions a set of objects into equivalence classes

Same as equivalence_partition() but also numbers the classes.

Args:
	iterable: collection of objects to be partitioned
	relation: equivalence relation. I.e. relation(o1,o2) evaluates to True
		if and only if o1 and o2 are equivalent

Returns: classes, partitions, ids
	classes: A sequence of sets. Each one is an equivalence class
	partitions: A dictionary mapping objects to equivalence classes
	ids: A dictionary mapping objects to the indices of their equivalence classes
"""
classes, partitions = equivalence_partition(iterable, relation)
ids = {}
for i, c in enumerate(classes):
	for o in c:
		ids[o] = i
return classes, partitions, ids


def check_equivalence_partition(classes, partitions, relation):
"""Checks that a partition is consistent under the relationship"""
for o, c in partitions.items():
	for _c in classes:
		assert (o in _c) ^ (not _c is c)
for c1 in classes:
	for o1 in c1:
		for c2 in classes:
			for o2 in c2:
				assert (c1 is c2) ^ (not relation(o1, o2))


def test_equivalence_partition():
relation = lambda x, y: (x - y) % 4 == 0
classes, partitions = equivalence_partition(
	range(-3, 5),
	relation
)
check_equivalence_partition(classes, partitions, relation)
for c in classes: print(c)
for o, c in partitions.items(): print(o, ':', c)


if __name__ == '__main__':
test_equivalence_partition()

a = {1, 2, 3, 6}

Third thing I learned:

Functions are what they call them first class objects, as they can be passed as arguments to other functions. However, functions, if not passed with the argument parenthesis are interpreted as objects (Duke.edu, 2024):

								
def function1(f):
    print(f"{f}")


def function2():
    function3 = lambda x: 3
    function1(function3("hello"))


if __name__ == "__main__":
    function2()

###OUTPUT: 3###

If we replaced function1function3("hello") by function1(function3) without the parenthesis, the output would be:

Output = <function function2.<locals>.<lambda> at 0x10dcf8d60>

Fourth thing I learned

I worked on the __next__ and __iterable__ methods for classes that want to be taken as for instance a group of objects and want to pass every object as if it was a list, without saving them in a list. This is how they are built:

								
class MyIterable:
    def __init__(self, data):
        self.data = data
        self.index = 0  # Initialize index to keep track of current position

    def __iter__(self):
        # Return an iterator object
        return self

    def __next__(self):
        # Check if the index is within bounds
        if self.index < len(self.data):
            # Get the current element and increment the index
            element = self.data[self.index]
            self.index += 1
            return element
        else:
            # If index is out of bounds, raise StopIteration
            raise StopIteration()


# Create an instance of MyIterable
my_iterable = MyIterable([1, 2, 3, 4, 5])

# Iterate over my_iterable using a for loop
for item in my_iterable:
    print(item)

class mrIteration:

def __init__(self, data):
	self.index = 0
	self.content = data

def __iter__(self):
	return self

def __next__(self):
	if self.index < len(self.content):
		element = self.content[self.index]
		self.index += 1
		return element
	else:
		raise StopIteration

def __getitem__(self, item):
	return self.content[item]

def pop(self):
	self.content = self.content[0:len(self.content) - 1]
	return self.content


b = mrIteration([{1}, {2}, {3}])

c = {}

for _ in range(-2, 2):
if _ < 0:
	c[_] = b[0]
elif 0 <= _ < 1:
	c[_] = b[1]
elif 1 <= _ < 3:
	c[_] = b[2]

print(c)
